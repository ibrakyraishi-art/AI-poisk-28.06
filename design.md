## Problem Statement

Ручной конкурентный анализ — дорого и медленно: маркетолог или продакт тратит дни на
сбор отзывов из App Store / Google Play, чтение новостей, составление матрицы сравнения.
Результат устаревает сразу после публикации. Нужна система, которая делает это за него.

Система принимает: название компании + период (`"30d"` / `"90d"` / `"6m"`) +
опциональный список конкурентов. Отдаёт: структурированный отчёт с матрицей
сравнения, стратегическими выводами и рекомендациями. Все данные с источниками (пруфами).

## What Makes This Cool

RAG-память как "умная доска" — агенты пишут, Manager читает, ничего не теряется
между шагами. Каждый агент получает свой мозг (LLM): дешёвый Gemini Flash для
сбора данных, умный Claude Opus для анализа. Пользователь видит не чёрный ящик,
а конфигурируемую команду.

"Вау"-момент: вводишь "FitTrack" — через 15-30 минут получаешь полный анализ с
цитатами из отзывов, ссылками на статьи и конкурентной матрицей. Можно поменять
мозг любому агенту и увидеть, как меняется качество.

## Constraints

- Стек: Python, CrewAI (hierarchical), Streamlit, ChromaDB (локально)
- Источники: Google Play (`google-play-scraper`), App Store (`app-store-scraper`),
  News API, веб-поиск через **SerperDev** (`serper-python`) — ключ в `.env`
- Выбор мозгов: dropdowns в Streamlit, модели `gemini-1.5-flash`, `claude-haiku-4-5-20251001`,
  `claude-sonnet-4-6`, `claude-opus-4-8`, `gpt-4o-mini` — маппинг в `config/llm_config.py`
- LiteLLM-префиксы в config: `gemini/gemini-1.5-flash`, `claude-haiku-4-5-20251001`,
  `claude-sonnet-4-6`, `claude-opus-4-8`, `gpt-4o-mini` (используются CrewAI для роутинга)
- Embedding model для ChromaDB: `all-MiniLM-L6-v2` через `sentence-transformers`
  (локально, без API-ключей); задаётся в `memory/vector_store.py` при создании коллекции
- Один разработчик, сайд-проект
- Streamlit — финальный UI (Этап 6+); консольный вывод приемлем для Этапов 1-5
- Скрейперы сторов нестабильны — принять как риск, добавить моки (`tests/fixtures/`) для тестов
- Стоп по насыщению Reviews Agent: `MAX_REVIEWS = 200`; батч = 50 отзывов;
  остановка если <5% семантически новых тем (cosine similarity <0.85 к уже сохранённым в ChromaDB)
- Каждый запуск создаёт изолированную ChromaDB-коллекцию (`fittrack_20260628`);
  старые коллекции чистятся через 7 дней или вручную через `memory/cleanup.py`
- ChromaDB документ-схема: `{text: str, metadata: {agent: str, run_id: str, source_url: str, timestamp: str}}`

## Premises

1. **CrewAI hierarchical** — правильный фундамент для обучения. Роли читаются как
   документация, понятно что делает каждый агент. [ПРИНЯТО]

2. **App Store / Google Play скрейпинг** — реальные данные + учёба на нестабильных
   инструментах. Риск падения API принят, добавить моки для тестов. [ПРИНЯТО]

3. **Streamlit-дашборд как основной вывод** — интерактивный UI + Word как экспорт
   по кнопке. Полноценно использует стек. [ПРИНЯТО]

## Approaches Considered

### Approach A: MVP-спринт
Один умный Research-агент (Reviews + News в одном) + Manager-проверка + ChromaDB in-memory.
Без выбора мозгов, без параллельности, без Word-экспорта.
Effort: S (1-2 недели) | Risk: Low
Плюс: работает за выходные. Минус: придётся рефакторить под полную архитектуру.

### Approach B: Staged build по плану (ВЫБРАНО)
Из 8 этапов в исходном документе (Part 8) реализуем 6 ключевых + опциональный
мониторинг. Manager + 7 агентов + RAG + выбор мозгов + Streamlit + Word-экспорт.
Каждый этап — рабочая версия.
Effort: XL (16-24 недели, сайд-проект) | Risk: Med

### Approach C: Streaming-first
Process.sequential + Python-оркестратор + стриминг агентов в Streamlit в реальном времени.
Без иерархического Manager, без ConditionalTask.
Effort: M (3-5 недель) | Risk: Med

## Recommended Approach

**Approach B: Staged build** — план уже написан (Part 7: все промпты готовы, Part 8:
8 этапов с чёткими результатами каждого). Каждый этап = рабочая точка останова.
Для обучения важно пройти полный цикл: RAG, иерархия, ConditionalTask, мониторинг.

Порядок этапов из документа (6 этапов + опциональный Этап 7):
1. Скелет + LLM config + Reviews Agent с лимитами (`Process.sequential`)
2. RAG-память (ChromaDB, коллекции-per-run) + News Agent (флаг `async_execution=True` проставляется, но реальная параллельность — в Этапе 4)
3. Analyst + Report Writer → `output/report.docx`; схемы: `ReviewData`, `AnalystOutput`, `ReportData`, `CompetitorProfile`
4. Manager + переход на `Process.hierarchical` + проверки через RAG (`ConditionalTask`); здесь `async_execution=True` начинает работать
5. Business Context + Competitor Finder (источник: SerperDev web search) + матрица сравнения
6. Streamlit UI: выбор мозгов, пресеты, кнопка скачать; `crew.kickoff()` в `threading.Thread`,
   результат через `queue.Queue`; UI обновляется через `st.empty()` + `time.sleep(1)` polling loop;
   **предупреждение о стоимости** перед запуском с Opus (≈$1-5 за прогон)

**Примечание:** Этапы 1-3 работают на `Process.sequential`. Этап 4 вводит `Process.hierarchical`.
`async_execution=True` на задачах — флаг CrewAI, который активирует параллельный диспатч
только под `Process.hierarchical`; в `Process.sequential` он игнорируется. Поэтому реальной
параллельности Reviews + News нет до Этапа 4 — это нормально, на Этапах 1-3 они работают последовательно.

**ConditionalTask пример:** Если `ReviewData.review_count < 10` или `confidence < 0.7`,
Manager возвращает Reviews Agent с расширенным периодом (+30 дней). Максимум 2 круга
(из `config/limits.py`). Реализация: callback function в `tasks/research_tasks.py`.
*Caveat:* ConditionalTask API нестабилен между версиями CrewAI — зафиксировать
`crewai==0.x.y` в `requirements.txt` и проверить community examples перед реализацией.

**Мониторинг (опционально, Этап 7):** Только `AgentOps` базовый dashboard (расходы,
токены, replay) без кастомного визуала. Добавлять только после того как все 6 этапов
работают стабильно.

## Success Criteria

- Этап 1: `python main.py` запускает Reviews Agent, собирает реальные отзывы FitTrack,
  выводит сгруппированный список тем с тональностью. LLM меняется через config.
- Этап 2: Reviews + News запускаются последовательно (параллельность — с Этапа 4),
  данные сохраняются в ChromaDB-коллекцию `{company}_{date}`,
  `memory_search('цена')` возвращает записи от обоих агентов.
- Этап 3: `output/report.docx` содержит матрицу сравнения и стратегические выводы.
- Этап 4: `Process.hierarchical` активен. Manager проверяет работу через RAG и
  возвращает агенту на доработку через ConditionalTask без вмешательства пользователя.
  Лог проверок виден в консоли.
- Этап 5: Competitor matrix JSON содержит минимум 2 конкурента с оценками по всем
  критериям (цена, рейтинг, UX, функции).
- Этап 6: Streamlit UI — ввод компании, dropdown выбора LLM для каждого агента
  (маппинг в `config/llm_config.py`), пресеты, кнопка "Анализировать",
  вывод отчёта + кнопка скачать Word.

CREATE TABLE embeddings (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  run_id     UUID REFERENCES analysis_runs(id) ON DELETE CASCADE,

CREATE INDEX ON embeddings USING hnsw (embedding vector_cosine_ops);

### Security Findings (Outside Voice)

Три реальные дыры, найденные независимым ревьюером:

**P0 — Нет проверки владельца job'а:**
`GET /analyze/{run_id}` не проверяет, что запрашивающий пользователь — владелец job'а.
Любой авторизованный пользователь может перебрать job_id и прочитать чужой отчёт.
Исправление: проверять `analysis_runs.user_id == JWT.sub` перед возвратом данных.

**P1 — Зависшие job'ы без watchdog'а:**
При падении Railway-контейнера (OOM, deploy) `crew.kickoff()` не завершается,
но `analysis_runs.status` остаётся "running" навсегда.
Исправление: cron каждые 10 мин — пометить "failed" все runs старше 45 мин.

**P1 — LLM-ключи в логах:**
Ключи из Vault передаются в CrewAI как обычные строки. Unhandled exception
с traceback может залогировать их. Vault решает проблему хранения, но не in-process exposure.
Исправление: `SecretStr`, `log_level=WARNING`, `--no-access-log` в prod.

### Failure Modes

| Сбой | Последствие | Митигация |
|------|------------|-----------|
| Scraper возвращает 0 отзывов | Reviews Agent без данных | Mock fixtures (`tests/fixtures/`) |
| SerperDev rate limit | Competitor Finder падает | Manager retry × 1; фиксировать в отчёте |
| Railway OOM/deploy mid-run | Status stuck at "running" | Watchdog cron каждые 10 мин |
| LLM-ключ пользователя просрочен | CrewAI throws mid-crew | Catch → write "failed" в analysis_runs |
| pgvector размерность не 384 | INSERT падает | Валидировать embedding.shape перед INSERT |
| ConditionalTask сломался после обновления crewai | Retry loops не работают | Pin `crewai==x.y.z` в requirements.txt |
| Supabase Storage upload falls | docx потерян | UI fallback на JSON-only view |
| Saturation stop не срабатывает | Crew читает >200 отзывов | Hard cap MAX_REVIEWS=200 |
